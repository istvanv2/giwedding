# WhatsApp link preview – causes and fixes

Why the same link can show a preview image for one person and not for another, and how to make sure tapping the preview opens the link correctly.

---

## 1. WhatsApp-side reasons

| Cause | What happens | What you can do |
|-------|----------------|------------------|
| **Preview disabled in privacy** | Recipient has “Disable link previews” in WhatsApp → Privacy. They only see the URL, no image. | Ask them to enable link previews in Settings → Account → Privacy. |
| **First-time glitch** | Previews sometimes fail to load the first time. | Send or paste the same link again; the image often appears. |
| **Network / firewall** | Corporate proxy, VPN, or strict firewall blocks WhatsApp from fetching the page. | Try on mobile data or another network; avoid blocking WhatsApp’s crawler. |

---

## 2. Website / metadata reasons

WhatsApp fetches the page once and reads Open Graph tags (`og:title`, `og:description`, `og:image`). If anything is wrong, the image (or whole preview) can be missing.

| Cause | What we did / what to check |
|-------|-----------------------------|
| **Missing or wrong `og:image`** | We use a single canonical OG image with an **absolute URL** (`OG_IMAGE_URL` → `og-preview.jpg`) in both root and locale layouts so crawlers always get a full URL. |
| **Image URL returns error or redirect** | Image is served from `/images/og-preview.jpg` (no redirect). Generate it with `npm run generate:og` from `public/images/1.jpg`. |
| **Image size/format** | We use a **smaller dedicated preview** (800×420, ~1.91:1) in `public/images/og-preview.jpg` for faster, more reliable loading. Run `npm run generate:og` after changing the hero image. |
| **Caching** | After changing the preview image, WhatsApp may still show the old one. We use **`?v=1`** on the image URL; when you change the image, bump to `?v=2` in `lib/site.ts` (`OG_IMAGE_URL`) so WhatsApp fetches the new image. |
| **Same link, different previews** | WhatsApp caches per URL. An old “no image” cache can persist. Sending a **slightly different URL** (e.g. `https://giwedding.com/ro?v=2` or `https://giwedding.com/en`) forces a new fetch and can make the image appear. |

---

## 3. Tap on preview – open link and work well

- **Link target**: The preview opens the **shared URL** (e.g. `https://giwedding.com/ro` or `https://giwedding.com/en`). We set `openGraph.url` to the same canonical URL for each locale so the preview “belongs” to the correct page.
- **No broken redirects**: Root path redirects to the default locale; `/ro`, `/en`, etc. render normally, so opening from WhatsApp goes straight to the right page.
- **Recommendation**: Share the **full URL with locale** (e.g. `https://giwedding.com/ro` or `https://giwedding.com/en`) so the preview and tap-to-open match the language.

---

## 4. What to test

1. **OG tags**: Use [Facebook Sharing Debugger](https://developers.facebook.com/tools/debug/) (or similar) and paste `https://giwedding.com/ro` (or `/en`). Check that `og:image` is absolute and the image loads.
2. **Image**: Ensure `public/images/og-preview.jpg` exists (run `npm run generate:og`). It’s 800×420 JPG for fast previews.
3. **After changing the image**: Run `npm run generate:og`, bump `?v=` in `lib/site.ts` (`OG_IMAGE_URL`), and if needed share a URL with a query param (e.g. `?v=2`) once to trigger a fresh preview fetch.
4. **Tap behaviour**: Share a link in WhatsApp, wait for the preview to load, then tap the preview and confirm it opens the correct locale page.

---

## 5. Summary of changes in this project

- **Absolute OG image URL** in `app/layout.tsx` and `app/[locale]/layout.tsx` via `lib/site.ts` (`OG_IMAGE_URL`), so crawlers always get a full URL.
- **Smaller preview image** (`og-preview.jpg`, 800×420) generated by `npm run generate:og` for faster, more reliable WhatsApp/social previews.
- **Cache-busting** on the image (`?v=1`); bump the version when you change the image.
- **Canonical and `openGraph.url`** use `SITE_URL` so preview and tap-to-open point to the same page.
